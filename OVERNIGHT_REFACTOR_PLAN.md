# Overnight Refactor — Game Loop & Health Plan

**Branch:** `feature/general-fixes`
**Working tree:** clean
**Goal:** Comprehensive single-player polish + architectural cleanup that doesn't paint MP into a corner. User is asleep; safe defaults everywhere; build stays green at every phase.

---

## Context

The user opened with two asks:

1. **Gitignore `.claude/worktrees/` and delete the tracked files.** Concrete cleanup item.
2. **Wide health scan + overnight refactor.** Take full control of the gameloop, fix errors and inconsistencies, structure the work so multiplayer is viable later.

Three audits were run in parallel (gameplay loop, code quality, architecture/MP-readiness). They converged on the same picture:

- **No critical runtime crash bugs.** Build is healthy.
- **Real game-feel bugs exist** (cargo cell miscount across ports, dialogue-while-walking, fragile player spawn, dead harbour-dues screen, shipwright doesn't validate before charging).
- **Persistence layer is captain-only** — accepted contracts, in-flight cargo, ship position, player on-foot position, throttle/rudder state all reset on reload. This is the biggest single-player feel gap.
- **Fuel station exists as decoration** — model loads, no gameplay mechanic behind it.
- **Architecture has four overlapping "player state" abstractions** (`PlayerSession`, `GameState`, `LocalPlayerView`, `PlayerVessel`). The MP seam (`LocalPlayerView`) was added but everything bypasses it.
- **Two interaction surfaces** (`BridgeInteractable`, `CaptainsChair`) have ~250 lines of near-identical code.
- **Asset paths are scattered** across 20+ files (hat models, pallet models, port buildings).
- **3 duplicate `_load_json` implementations** (`ship_builder.gd`, `mesh_transformer.gd`, `model_assembler.gd`).
- **`.claude/worktrees/`** committed by accident from prior agent sessions.

The plan below is structured into phases. Each phase is **a single commit**, **verifiable in isolation**, and **leaves the build playable**. Order is chosen so early phases unblock later ones and the architectural unification (Phase B) lands before the features that depend on it (Phases C-F).

## Phase 1 — Quick correctness fixes (small, surgical)

**Risk:** low. Each fix touches one file, one function.

These are all confirmed bugs from the audit. Bundling them into one commit because individually they're tiny.

### 1.1 Cargo cell miscount across ports

`scripts/npc/contract_npc.gd::_ship_cells_free()` (and any sibling that counts free cells) currently counts cells where `CargoDeckComponent.port_id.is_empty()`. A ship floating in open water (unmoored) reports all cells as free; a player can over-commit to two ports' worth of cargo.

**Fix:** filter on **the deck's `port_id` matching the contract's `origin_port_id`, OR the deck being moored at the same port as the contract NPC.** Get current mooring state from `MooringComponent` or `PortDock`.

Existing infra to reuse: `MooringComponent._sync_berth_with_dock()` already sets `port_id` on the deck when moored. So we can require `deck.port_id == this_npc.port_id` for cells to count as available.

### 1.2 NPC dialogue input suppression

`scripts/npc/npc_interactable.gd::_on_interact()` opens a dialog but doesn't stop the player from walking. The player can wander away mid-conversation.

**Fix:** on `open_ui()`, call `player.set_process_unhandled_input(false)`; on `close_ui()`, restore. Or — cleaner — set `Input.set_mouse_mode(VISIBLE)` and route the dialog's `process_mode` so it pauses gameplay. Either approach works; pick the lighter one (input-suppression flag on the player node).

Existing infra to reuse: `CaptainsChair._board()` already does `_player.set_process_unhandled_input(false)`. Mirror that pattern.

### 1.3 Player spawn safety

`scripts/world/world.gd::_spawn_player()` falls back to `Vector3(0, 0.5, 0)` if `PortPlot.get_spawn_position()` returns null. No collision validation.

**Fix:** after computing the spawn position, validate it with a tiny `PhysicsDirectSpaceState3D.intersect_point` query (player capsule, layer "world"). If it intersects, raise Y by `0.5` and retry up to 3 times. As ultimate fallback, snap to the home port's `PortDock.get_spawn_position()` even if `get_spawn_position()` returned something odd.

### 1.4 Wire `distance_sailed_m`

`PlayerSession.add_distance_sailed()` exists but is never called.

**Fix:** in `BoatBody._physics_process()` (or `BoatController`), integrate `linear_velocity.length() * delta` and emit to `PlayerSession.add_distance_sailed()` once per second (not per physics tick — debounce). Only when the boat is helmed by the player, to avoid counting AI ship motion.

Existing infra to reuse: `LocalPlayerView._helmed` already tracks the helmed boat. Subscribe to its `helm_changed` and integrate on the active boat.

### 1.5 Remove dead PAY_DUES screen

`scripts/npc/harbour_master_npc.gd::_Screen.PAY_DUES` exists and always returns "no dues."

**Fix:** delete the enum value, the screen handler, and the button that leads there. Trim ~30 lines.

### 1.6 Shipwright pre-charge validation

`scripts/npc/shipwright_npc.gd::_try_pay_for_commission()` charges the player marks *before* checking the hull JSON exists. If `HullStations.from_hull_json(null, ...)` crashes, the player is charged and gets nothing.

**Fix:** validate `FileAccess.file_exists(hull_path) and ShipBuilder._load_json(hull_path).has("parts")` before deducting marks. Refund-by-not-charging is the right pattern.

### 1.7 Mooring split-berth UI feedback

`scripts/ship/mooring_component.gd::_mooring_splits_berths()` rejects mooring with no UI feedback.

**Fix:** on rejection, emit a signal like `mooring_rejected(reason: String)`. The ShipHud can render a short flash message ("Lines on different berths — release and retry").

Existing infra to reuse: `ShipHud` already draws status badges (lights, wind). A flash overlay is a small addition.

**Verify Phase 1:** boot the world. Walk into NPC and confirm you can't walk away mid-dialogue. Accept a contract at home, sail to a second port without delivering, try to accept a contract — should reject with proper free-cell count. Sail near a long dock and tie lines across two berths; HUD shows rejection.

## Phase 3 — `LocalPlayerView` as the single source of truth

**Risk:** medium. Touches a lot of UI but each migration is small.

The MP seam exists; nothing uses it. Make it the canonical lookup for "what is the player doing" and fold `PlayerVessel` into it as an internal helper.

### 3.1 Fold `PlayerVessel` group-tagging into `LocalPlayerView`

`PlayerVessel` (RefCounted static) tags ships with "player_boat" and despawns old hulls. Move these into `LocalPlayerView` methods:

```
LocalPlayerView.mark_active_ship(ship)
LocalPlayerView.despawn_other_ships(except=null)
LocalPlayerView.unregister_ship_from_docks(ship)
```

Old `PlayerVessel` becomes a thin forwarding shim for one commit (back-compat); next commit removes it.

### 3.2 Migrate `GameState` to read through `LocalPlayerView`

`GameState._wire_player_session()` and `_wire_contract_registry()` currently grab `/root/PlayerSession` and `/root/ContractRegistry` directly. Replace with `LocalPlayerView.session()` / `LocalPlayerView.registry()` accessors.

Keep `GameState` — it remains the read model for sub-state signals. Internal source switches.

### 3.3 Migrate `ShipHud` and `WalkingHud` and NPC interactables

Every `get_node_or_null("/root/PlayerSession")` in UI and NPC code becomes `get_node_or_null("/root/LocalPlayerView")`. Strict rule: **no UI code directly accesses PlayerSession or ContractRegistry from this commit onward.**

`WalkingHud` already partially migrated; finish it.

Files: `scripts/ui/ship_hud.gd`, `scripts/ui/walking_hud.gd`, `scripts/ui/main_menu.gd`, `scripts/ui/game_menu.gd`, `scripts/ui/debug_draw.gd`, `scripts/npc/harbour_master_npc.gd`, `scripts/npc/shipwright_npc.gd`, `scripts/npc/contract_npc.gd`.

### 3.4 Drop `PlayerVessel.gd`

Remove the shim. All callers now go through `LocalPlayerView`.

**Verify Phase 3:** boot world, commission a ship, helm it, sail, dock, talk to all three NPCs. HUDs update correctly. Marks display, contracts accept, ship commissions all still work — but every read passes through `LocalPlayerView`.

## Phase 4 — Persistence completeness

**Risk:** medium. New data fields in `PlayerData` need version-bump handling.

### 4.1 Bump save format version + migration

`PlayerSaveStore` envelope is `{ version: 1, player: {...} }`. Bump to version 2. Add a `migrate_v1_to_v2(envelope)` step that fills new fields with safe defaults so existing saves keep loading.

### 4.2 Persist accepted contracts

`PlayerData.accepted_contracts: Array[Dictionary]` — each entry `{contract_id, taken_count, delivered_count}`.

On save: snapshot `ContractRegistry.get_accepted_contracts()` into this array.

On load: `ContractRegistry.replay_accepted(records)` re-calls `accept_contract()` per record with the saved counts. This replays the *intent* without re-spawning pallets — pallets are restored separately (4.4).

### 4.3 Persist ship runtime state

`PlayerData.active_vessel` currently has `{uid, hull_id, display, template_path}`. Add:

```
ship_position:    Vector3
ship_rotation_y:  float  (yaw only, full quat is overkill for surface boats)
throttle_stage:   int
rudder_angle:     float
fuel_pct:         float  (placeholder, filled in Phase 5)
```

Save: on quit, helm-exit, and periodic auto-save (every 60s of wall-clock, debounced via `_request_save()`).

Load: after harbour master spawns the ship, restore position/rotation/throttle/rudder before the player boards.

### 4.4 Persist cargo manifest

`PlayerData.active_vessel.cargo: Array` of `{deck_name, cell_idx, contract_id, units}`.

Save: walk every `CargoDeckComponent` under the player's ship, serialize each pallet's contract reference + position.

Load: after spawn, walk the saved cargo records and re-add pallets to the corresponding decks. Uses existing `CargoDeckComponent.add_pallet()` path.

### 4.5 Persist on-foot player position

`PlayerData.player_position: Vector3`, `player_port_id: String`.

Save: whenever player is on foot (not helming), record position every 60s.

Load: if `player_port_id` matches an existing port, spawn the player at the saved position; if it doesn't, fall back to home port spawn.

**Verify Phase 4:** start new game, accept a contract, pick up cargo, sail halfway to the destination, quit. Relaunch — contract should still be accepted, cargo still on board, ship at the saved position with the saved throttle setting. If the player was on foot in a port, they wake up where they left off.

## Phase 5 — Fuel system MVP

**Risk:** medium. Touches several systems but mechanic is well-scoped.

### 5.1 Fuel tank on `BoatBody`

Add `@export var fuel_capacity_l: float` and `var fuel_l: float`. Capacity scales with hull (e.g. `displacement_kg * 0.01`). Filled to capacity on commission.

Methods: `consume_fuel(litres)`, `add_fuel(litres)`, `get_fuel_pct() -> float`.

### 5.2 Consumption in `PropulsionComponent`

In `_physics_process`, integrate `|throttle| * burn_rate_l_per_s * delta` and deduct from the parent BoatBody's `fuel_l`. Cap throttle at 0 if tank is dry (engine stalls). Burn rate per-ship from a hull-data field (added to hull JSON, with a sensible default if missing).

### 5.3 Refuel interaction at the pump

`FuelStation` currently just draws a model. Add an interactable mesh near `get_pump_position()` (use a small `Area3D` + `BoardingInteractable`-style prompt). On interact while standing next to it and a moored player ship is in range, transfer fuel from infinite station tank to the ship at a cost in marks (e.g. `0.1 marks / litre`). Use `LocalPlayerView.spend_marks()`.

### 5.4 Fuel gauge in `ShipHud`

Add a vertical fuel bar next to the throttle telegraph. Colour-coded: green > 30%, amber 10-30%, red < 10%. Flash on dry tank.

### 5.5 Out-of-fuel state

When `fuel_l == 0`, force throttle to 0 (`PropulsionComponent.max_thrust * 0` effective). HUD shows "DRY TANK — DRIFTING". Player still has rudder authority for current/momentum-driven steering but no thrust.

**Verify Phase 5:** start a new game, commission a ship at the shipwright, board it. Throttle up, sail until the fuel runs out (or burn-rate-cheat for testing), confirm propulsion stops. Sail/drift back to a port, dock, walk to the fuel pump, refuel, return to helm — fuel restored, propulsion resumes.

`ContractRegistry._export_stock` depletes but never replenishes. Add a periodic restock tied to `WorldClock`:

```

    for commodity in port.exports:

```

Hook into `WorldClock.game_hour_changed` (or add that signal if missing).

## Phase 7 — Abandon-ship + small UX polish

**Risk:** low.

### 7.1 Abandon ship mechanic

Add a captain's chair option or `BoatController` key (`Esc` while helming → "Exit helm" already exists, so a separate `Shift+Esc` → "Abandon vessel"). On confirm:

1. Forfeit all cargo on board (existing `LocalPlayerView.despawn_other_ships` already does this — call it directly).
2. Despawn ship.
3. Respawn player at the home port.
4. Clear `active_vessel`; harbour master will offer the starter vessel again.

Confirms via dialog ("Abandon vessel? This forfeits all cargo and your current ship.").

### 7.2 Water rescue robustness

`Player._process` already teleports to `_last_safe_position` when below `abyss_reset_y`. Two improvements:

- Update `_last_safe_position` whenever the player crosses a `PortDock`'s `port_id` zone, not just `is_on_floor()`. This catches the "ran off the end of a long dock" case.
- Brief HUD flash when water rescue fires ("Pulled to shore").

### 7.3 Per-port spawn anchor metadata

Right now home port spawns the player at `PortDock.get_spawn_position()`. For consistency when loading saves with `player_port_id` (Phase 4.5), add `PortDock.get_safe_spawn_position()` that returns a guaranteed-safe spot (above the deck, inside the bounding box of the port plot, with collision validation).

**Verify Phase 7:** accept a contract, sail somewhere, hit Shift+Esc — confirm abandonment dialog, ship despawns, player back at home port, cargo gone, contract reset to available. Walk off the end of the dock — water rescue catches you and HUD flashes.

## What's deliberately out of scope

- **Multiplayer transport.** The user wants MP later but this plan is single-player polish.

- **Death of the player character.** Water rescue is the existing mechanic; we improve it but don't add a death state.
- **Visual asset upgrades.** The shipping art style (hand-modeled JSONs, vertex colours, no textures) is intentional; not touching it.
- **Map overlay further split.** Already shipped in PR #12; current 652-line state is fine.

## Order rationale

- **Phase 0** is the explicit user ask + a one-line clean-up.
- **Phase 1** fixes confirmed bugs in isolation so later refactors don't conflate fixes with restructure.
- **Phase 2** removes duplication so Phase 3's migration has fewer places to touch.
- **Phase 3** establishes `LocalPlayerView` as the seam *before* Phase 4 adds persistence — saves go through the right interface from day one.
- **Phase 4** is the biggest single-player feel improvement — quitting and resuming actually works.
- **Phase 5** is the biggest missing mechanic — fuel makes the world feel less arbitrary.
- **Phase 6** makes the gameplay loop visible to the player (cargo journal) and self-sustaining (restock).
- **Phase 7** closes the rough-edges polish gap.
- **Phase 8** is the docs refresh that locks in the architectural decisions.

Each phase ships as its own commit with a clear scope. If anything regresses, the bad phase is identifiable and revertable in isolation.
