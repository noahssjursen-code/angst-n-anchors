# Ship Fleet Expansion — Agent Handoff

Handoff document for continuing shipwright refinement and expanding beyond bulk-cargo freighters (fishing boats, ferries, etc.). Written from design review and codebase audit on branch `feature/shipwright-refinement` (May 2026).

**Related docs:** [`SHIP_BUILDING_OVERHAUL.md`](SHIP_BUILDING_OVERHAUL.md), [`SHIP_BUILDING.md`](SHIP_BUILDING.md), [`ARCHITECTURE.md`](ARCHITECTURE.md), [`AGENTS.md`](AGENTS.md)

**Dev scene for visual review:** `scenes/ship_showcase.tscn` + `scripts/ship/ship_showcase.gd` — grid of all hull variants with length-aware spacing, turntable spin, price labels, auto-framing camera.

---

## 1 — Executive summary

The current fleet is **one cargo-ship pipeline at multiple scales**. All hulls share the same mesh topology (scaled prism), bridges are scaled box kits (`tools/gen_bridges.py`), and extended variants often reuse the same superstructure on a longer hull. Players perceive “10 sizes of the same boat,” not distinct vessel types.

To add fishing boats, ferries, tugs, etc., the codebase needs:

1. **Visual differentiation** within the existing cargo progression (Phase 1–3 below).
2. **Architectural separation** of **size** (`ShipClass`) vs **role** (`VesselRole`) vs **specific hull** (`hull_id`).
3. **A central hull registry** so catalog data is not duplicated across NPCs, replication, and showcase code.
4. **Capability-driven assembly** in `ShipBuilder` so not every hull gets cargo decks.

**Colors:** Identical fleet-wide — black upper, red keel, gray deck. No per-class livery.

**Location:** `resources/data/models/superstructures/` — **5 active bridge types** (+ 2 unused legacy: `bridge_small.json`, `bridge_medium.json`)

| Bridge file | House size (W×H×D m) |
|-------------|----------------------|
| `bridge_coastal_trader.json` | 2.2 × 2.3 × 1.8 |
| `bridge_cargo_ship.json` | 3.4 × 3.2 × 3.8 |

| `bridge_handysize_feeder.json` | 5.0 × 3.5 × 3.0 |

Extended catalog entries (`*_long`, `large_freighter`) **reuse the base-class bridge** on a longer hull → superstructure looks proportionally small / identical.

### 2.3 Catalog and duplication

Hull catalog entries are **hardcoded in multiple places**:

| Location | Purpose |
|----------|---------|
| `scripts/npc/shipwright_npc.gd` → `HULL_CATALOG` | 8 retail hulls, shipwright UI |
| `scripts/ship/starter_vessel.gd` → `ENTRY` | Free 13 m starter (harbour master) |
| `scripts/network/replication_drawing_service.gd` → inline `ship_catalog` dict | Remote ship visual spawn |
| `scripts/ship/ship_showcase.gd` | Reads both sources for dev grid |

**9 shipwright entries + 1 starter = 10 playable hulls.** Grouped by display label `ship_class_label` (Coastal Trader, Short Sea Coaster, etc.) — this is **marketing copy**, not a gameplay enum.

### 2.4 Assembly pipeline

```
Catalog entry / template JSON
   ↓
ShipBuilder.build(template_path) → BoatBody
   ├── ModelAssembler (hull JSON)
   ├── Superstructure at slots["bridge"] + SUPERSTRUCTURE_OFFSET
   ├── StripBuoyancy, Hydrodynamics, Propulsion, Rudder, BowThruster
   ├── MooringComponent + bollards (scale with hull length)
   ├── Hull lights from JSON
   └── CargoDeckComponent(s) from hull cargo_decks — ALWAYS for cargo hulls
```

Preview path (`scripts/npc/shipwright_preview.gd`) mirrors hull + bridge only (no gameplay components).

**Key finding:** `ShipBuilder` always calls `_add_cargo_decks()`. There is no branching for non-cargo vessel types.

### 2.5 Size classification (exists, role classification does not)

**File:** `scripts/ship/ship_class.gd`

`ShipClass.Type` enum: `LAUNCH`, `COASTAL_TRADER`, `SHORT_SEA_COASTER`, `HANDYSIZE_FEEDER`, `DEEP_SEA_FREIGHTER`

Used by:
- `scripts/port/port_dock.gd` — berth count, max vessel size, dock fit
- Contract UI hints (`CARGO_CELLS` per class)

Comments already mention “small ferries” under `LAUNCH` and “fishing vessels” under `COASTAL_TRADER`, but **no code branches on vessel role**.

### 2.6 Persistence (Go server)

**Table:** `vessels` — `hull_id VARCHAR(64)`, `display_name`, `fuel_litres`, `hull_health`, `template_path`, `cargo_manifest_json`

**Endpoints:** `GET/POST/DELETE /v1/vessels` in `angst-n-anchors-mp/internal/transport/httpapi/server.go`

`hull_id` is sufficient for now; role/class metadata can live in a client-side registry until a shared hull manifest is needed server-side.

### 2.7 What visual differentiation exists today

- 1 vs 2 logical cargo decks (gameplay grid only — **not visible deck geometry**)
- `hull_cargo_ship.json` has twin cargo deck layout (unique among coastal)
- Mooring bollard size scales with hull length
- No visible hatches, cranes, kingposts, or deck hardware on hull mesh

## 3 — Problem statement

### 3.1 Visual sameness (cargo fleet)

Players choosing a ship in the shipwright see **size differences only**. Root cause: procedural scaling (`gen_bridges.py` + hand-authored scaled hull vertices) optimized pipeline consistency over silhouette variety.

### 3.2 Single-profession assumption

The game is architected as **maritime cargo trading**. Adding fishing, ferries, passenger routes, etc. requires a second axis (role) without breaking berths, contracts, replication, or `ShipBuilder`.

---

## 4 — Recommended architecture: three axes

| Axis | Answers | Godot type | Example |
|------|---------|------------|---------|
| **ShipClass** | “Does it fit this berth/dock?” | `ShipClass.Type` enum (exists) | Ferry and coaster both `COASTAL_TRADER` |
| **VesselRole** | “What gameplay is this for?” | **New** enum or string | `CARGO`, `FISHING`, `PASSENGER`, `TUG` |
| **HullDefinition** | “Which specific build?” | Registry entry by `hull_id` | `fishing_skiff`, `coastal_trader_long` |

**Rule:** Size and role are independent. A 12 m fishing skiff and a 13 m coastal trader can share `ShipClass.COASTAL_TRADER` (or `LAUNCH`) but must differ in role, silhouette, components, vendor, and contracts.

---

## 5 — Phase A: Visual differentiation (cargo fleet, no new roles)

Do this **before or in parallel with** fleet expansion. High impact, lower architectural risk.

### A1 — Quick wins

1. **Per-class livery** — palette swaps in hull + bridge JSON (coastal lighter, deep sea darker, taller funnel, optional second mast on long variants).
2. **Fix extended-variant bridge proportions** — `*_long` hulls need scaled-up bridge JSON or a `_long` bridge variant, not the same bridge as the base hull.
3. **One signature feature per class** — e.g. cargo ship: visible twin houses; handysize: kingpost; deep sea: sharper bow / bulbous wedge in hull vertices.

### A2 — Silhouette identity (4 hull families)

Stop scaling one topology for everything. Target **4 distinct cargo hull shapes**:

| Class | Silhouette cue |
|-------|----------------|
| Coastal | Short bow, low freeboard, compact bridge forward |
| Short sea | Fuller bow, single cargo deck, moderate house |
| Handysize | Boxier midsection, deck equipment |
| Deep sea | Long straight run, aft superstructure, taller funnel |

Extended variants = same family, more length. **`hull_large.json` vs `hull_deep_sea_freighter_long.json`** should differ in beam or deck layout, not be near-duplicates at 60 m.

### A3 — Deck dressing

Slot-driven props from JSON: hatch covers, bollard clusters, winches, optional crane on cargo ship. Reuses `ModelAssembler` + hull `slots` — no new pipeline.

### A4 — Validation

Use `scenes/ship_showcase.tscn` to review all variants side-by-side. A class “reads” when identifiable at a glance in the grid.

## 6 — Phase B: Multi-type fleet architecture

### B1 — Central hull registry (do first)

Create `scripts/ship/hull_registry.gd` (or `resources/data/hull_registry.json` loaded by a thin GDScript wrapper).

**Single source of truth** for all hull definitions. Replace duplicated catalogs in:

- `scripts/npc/shipwright_npc.gd`
- `scripts/ship/starter_vessel.gd` (reference registry entry, keep template-writing logic)
- `scripts/network/replication_drawing_service.gd`
- `scripts/ship/ship_showcase.gd`

**Proposed entry shape:**

```gdscript
{
  "id": "fishing_skiff",
  "display": "Inshore Skiff • 8 m",
  "role": "fishing",                    # VesselRole
  "ship_class": ShipClass.Type.LAUNCH,  # berth/dock size
  "ship_class_label": "Inshore",        # UI grouping (optional)
  "hull_file": "hull_fishing_skiff.json",
  "superstructure": "wheelhouse_small",
  "vendor": "boatyard",                 # who sells it
  "capabilities": ["fish_hold", "mooring"],
}
```

**Registry API (suggested):**

- `get_by_id(hull_id) -> Dictionary`
- `all() -> Array`
- `by_vendor(vendor_id) -> Array`
- `by_role(role) -> Array`
- `by_ship_class(class) -> Array`

### B2 — VesselRole enum

Add `scripts/ship/vessel_role.gd`:

```gdscript
class_name VesselRole
enum Type { CARGO, FISHING, PASSENGER, TUG, WORKBOAT }
```

Use for: contract filtering, NPC catalog filtering, component attachment, UI copy.

### B3 — Capability-driven ShipBuilder

Today: `_add_cargo_decks()` always runs.

Target: hull JSON and/or registry entry declare capabilities; `ShipBuilder` attaches matching components:

| Capability | Component | Hull JSON key |
|------------|-----------|---------------|
| Cargo | `CargoDeckComponent` | `cargo_decks` (exists) |
| Fishing | `FishHoldComponent` (new) | `fish_hold` |
| Passenger | `PassengerDeckComponent` (new) | `passenger_decks`, optional `vehicle_ramp` |
| Mooring | `MooringComponent` | always / explicit |

Non-cargo hulls simply omit `cargo_decks`. Do **not** assume every vessel is a freighter.

### B4 — Vendor separation (not one mega-catalog)

| Vendor NPC | Sells | Filter |
|------------|-------|--------|
| **Shipwright** | Bulk cargo progression | `vendor == "shipwright"` |
| **Boatyard / Chandler** | Fishing, workboats, tenders | `vendor == "boatyard"` |
| **Ferry office** | Passenger vessels, route leases | `vendor == "ferry_office"` |
| **Harbour master** | Starter loaner | existing `StarterVessel` flow |

Each NPC uses the same catalog panel pattern (`ShipwrightCatalogPanel`) filtered by registry — avoid duplicating UI logic.

### B5 — Contracts keyed to role

Extend contract data (not just `ShipClass` / cargo cells):

```json
{ "required_role": "cargo", "min_cells": 12 }
{ "required_role": "fishing", "target_species": "cod" }
{ "required_role": "passenger", "min_berths": 40 }
```

`ShipClass` gates **where** you dock; `VesselRole` gates **what jobs** you accept.

**File to extend:** `scripts/npc/contract_npc.gd` and contract data resources.

### B6 — Visual strategy per role (new hull families, not scaled cargo prism)

| Role | Silhouette cues |
|------|-----------------|
| **Fishing** | Low freeboard, open aft deck, small wheelhouse, outriggers / net booms |
| **Ferry** | Box hull, side doors, upper passenger deck, vehicle ramp aft |
| **Cargo** | Current progression (once differentiated per Phase A) |
| **Tug / launch** | Short, wide, heavy superstructure forward |

Each role needs its **own hull JSON topology**, not a scaled copy of `hull_coastal_trader.json`.

### B7 — Database / multiplayer

No schema change required immediately. `vessels.hull_id` + client registry is enough.

Future: shared `hulls.json` manifest on server for validation; optional `role` column if server-side contract enforcement is needed.

Replication already spawns ships from `hull_id` → template; point lookup at `HullRegistry` instead of inline dict in `replication_drawing_service.gd`.

## 8 — What NOT to do (yet)

- **10 fully unique hand-modeled ships** — too much art cost before architecture is proven.
- **Full procedural hull generation** — evolve the existing JSON + `ModelAssembler` pipeline class-by-class.
- **Cosmetic-only randomization before class identity** — random paint on identical shapes still feels samey.
- **One shipwright catalog with every vessel type** — breaks fiction and UI clarity.
- **New `ShipClass` enum value per role** — ferries and cargo coasters share size tiers; use `VesselRole` instead.
